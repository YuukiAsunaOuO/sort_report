## 一、排序法介紹
### 1. 冒泡排序 (Bubble Sort)
原理：重複走訪要排序的數列，一次比較兩個元素，如果他們的順序錯誤就把他們交換過來。
### 2. 快速排序 (Quick Sort)
原理：選定一個基準值 (Pivot)，將數列分成比基準小和比基準大的兩部分，再遞迴排序。

| 演算法 | 平均時間複雜度 | 空間複雜度 | 穩定性 |

## 五、心得與結論
1. 效能分水嶺的觀察在實驗過程中，我發現 $O(n^2)$ 演算法（如 Bubble Sort）與 $O(n \log n)$ 演算法（如 Quick Sort）之間存在極其明顯的「效能斷層」。小數據規模 ($n < 500$)： 各演算法的執行時間都在毫秒等級，差異幾乎可以忽略。在這種情況下，實作簡單、程式碼量小的插入排序（Insertion Sort）反而具有優勢。大數據規模 ($n > 10,000$)： $O(n^2)$ 的演算法耗時呈指數級成長，甚至會導致程式當機或執行過久；而 Quick Sort 與 Merge Sort 依然能保持穩定的線性成長速度。這印證了理論複雜度在大數據時代的重要性。2. 理論與實驗的對照實驗結果基本符合 $O(n \log n)$ 優於 $O(n^2)$ 的理論預期，但仍有一些細節值得探討：快速排序 (Quick Sort) 的效率： 在隨機數據下，Quick Sort 往往是所有演算法中最快的，這歸功於它優異的快取記憶體利用率。插入排序 (Insertion Sort) 的驚喜： 當數據接近「幾乎已排序」的狀態時，Insertion Sort 的表現甚至優於進階排序法，這說明了演算法沒有絕對的好壞，只有「適不適合場景」。3. 穩定性與空間成本的權衡除了速度，實驗也讓我思考了其他維度：合併排序 (Merge Sort) 雖然在最差情況下依然穩定，但它需要額外的 $O(n)$ 空間來存放暫存陣列。在記憶體受限的環境中，這可能是個缺點。穩定性 (Stability) 的重要性：在處理複雜對象（例如依據「分數」排序，但希望「學號」維持原順序）時，穩定的排序法（如 Merge Sort）比不穩定的（Quick Sort）更具實務價值。4. 總結與反思透過這次 GitHub 報告的製作，我不僅練習了 Markdown 的排版與 Git 的使用，更深刻體會到「演算法是程式的靈魂」。在硬體效能突飛猛進的今天，我們不能只依賴硬體暴力運算。從 $O(n^2)$ 最佳化到 $O(n \log n)$，在處理百萬級數據時，節省的是數小時甚至數天的時間。這堂課讓我學會了在開發前先評估複雜度，選擇最適合當前場景的工具，這才是軟體開發者的核心競爭力。
